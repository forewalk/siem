## 1. 코드 품질 검토

### 1.1 코드 스타일 및 가독성
- [ ] 코드가 PEP 8 스타일 가이드를 따름
- [ ] 변수/함수/클래스명이 명확하고 일관성 있음
- [ ] 불필요한 주석이나 디버그 코드가 없음
- [ ] 코드 중복이 최소화됨
- [ ] 복잡한 로직에 적절한 주석이 있음

**의견:**


---

### 1.2 아키텍처 준수
- [ ] 레이어 구조(Model → Schema → Repository → Service → Endpoint)를 올바르게 따름
- [ ] 각 레이어의 책임이 명확히 분리됨
- [ ] Repository에만 데이터베이스 쿼리가 존재
- [ ] Service에만 비즈니스 로직이 존재
- [ ] Endpoint는 요청/응답 처리만 담당

**의견:**


---

### 1.3 네이밍 컨벤션 준수
- [ ] 테이블명: snake_case, plural (예: `users`, `order_items`)
- [ ] 컬럼명: snake_case, 명확한 의미 (예: `created_at`, `is_active`)
- [ ] Boolean 컬럼: `is_`, `has_`, `can_`, `should_` 접두사 사용
- [ ] 타임스탬프: `created_at`, `updated_at`, `deleted_at`
- [ ] Foreign Key: `{table_singular}_id` (예: `user_id`, `order_id`)

**의견:**


---

### 1.4 에러 처리
- [ ] 모든 예외 상황이 적절히 처리됨
- [ ] 에러 메시지가 명확하고 유용함
- [ ] 민감한 정보가 에러 메시지에 노출되지 않음
- [ ] 적절한 HTTP 상태 코드 사용 (400, 401, 404, 500 등)
- [ ] 로깅이 적절히 구현됨
